Anatomia dobrego prompta

🟠 ROLA - "Jesteś senior data scientistem w Allegro..."
🔵 ZADANIE - "Napisz SQL który policzy GMV netto per vertical..."
🟢 KONTEKST - "Tabela: General.orders, partycja: _partitiondate, okres: Q1 2026..."
🟡 FORMAT - "Odpowiedź jako SQL z CTE, komentarze po polsku..."
🟣 PRZYKŁAD (opcjonalnie) - "Wzorzec: SELECT ... FROM ... WHERE ... GROUP BY ALL"
Minimum:
ROLA + ZADANIE + KONTEKST
Bonus:
FORMAT + PRZYKŁAD → wyraźnie lepsza jakość

</kontekst>

<formatowanie>

Techniki promptowania

🧠 Chain of Thought

"Pomyśl krok po kroku"
Model rozkłada problem na etapy.
→ złożone analizy, debugging

🛑 Don't code yet

"Nie pisz kodu - omówmy architekturę"
AI zadaje pytania zanim napisze.
→ duże projekty, nowe domeny

📐 Zero-shot

Prosty prompt bez przykładów.
Działa, gdy model ma podpięte instrukcje.
→ "Policz GMV za wczoraj."

📋 Few-shot

Dajesz przykład rozwiązanego problemu wraz z tokiem rozumowania.
Model naśladuje i generuje podobne kroki.
→ Dane, insight, akcja

🔪 Problem decomposition

"Rozłóż zadanie na mniejsze kroki"
Duży SQL → osobne CTE → review.
→ pipeline'y, ETL

🔄 Self-refinement

"Oceń swój wynik 1–10 i ulepsz"
AI poprawia własne błędy.
→ iteracyjna poprawa jakości

Zaawansowane: self-refinement & ensembling

🔄 Self-refinement

Prompt → Wynik v1

↓ "Oceń wynik 1-10. Co poprawić?"

→ Wynik v2 (lepszy)

↓ "Zmień styl na bardziej zwięzły"

→ Wynik v3 (finalny)

Self-consistency: odpytaj ten sam model 3× - wybierz odpowiedź najczęstszą; spawnowanie subagentów.

🎲 Ensembling

Ten sam prompt →

Model A → wynik A

Model B → wynik B

Model C → wynik C

↓ Porównaj → wybierz najlepszy

Cross-model: Sonnet vs GPT vs Gemini - porównaj na tym samym zadaniu
💡 Szczególnie wartościowe przy ważnych decyzjach biznesowych.

Gdzie pisać prompta?

📄 Plik .md

✓ Prompt wielokrotnego użytku
✓ Złożony, wieloliniowy
✓ Chcesz iterować i ulepszać
✓ Chcesz wersjonować (Git)
Np. brief analizy, PRD, opis taska

💬 Okno czatu

✓ Jednorazowe pytanie
✓ Szybka interakcja
✓ Dialog
✓ Eksploracja tematu
Np. "do czego służy biblioteka pandas?"
─── METAPROMPTING ───
Prompt który generuje prompt. Przykład: "Zaprojektuj prompta do tworzenia raportów tygodniowych. Prompt powinien pytać o: metryki, okres, grupę docelową. Zapisz wynik w pliku weekly-report.prompt.md"
Kiedy? • Powtarzalny workflow (EDA, DAG, dashboard) • Standaryzacja w zespole • Budowa skills/agentów (sekcja 4)

Globalne vs lokalne — gdzie co leży

🌍 GLOBALNE (działają WSZĘDZIE)

C:\Users\<user>\
  AppData\Roaming\Code\
    User\prompts\
      ├── ds-core
      │   .instructions.md
      ├── sql-catalog
      │   .instructions.md
      ├── allegro_theme.py
      ├── *.prompt.md
      └── *.agent.md

Lub: ~/.github/prompts/

📁 LOKALNE (tylko w tym folderze)

moj-projekt/
├── .github/
│   ├── copilot-
│   │   instructions.md
│   ├── prompts/
│   │   └── *.prompt.md
│   └── skills/
│       └── <skill>/
│           └── SKILL.md
└── .vscode/
    └── settings.json
💡 Globalne = Twój styl pracy (wszędzie taki sam)
   Lokalne = specyfika projektu (inne tabele, inne metryki)
⚠️ Prompt stack: System Prompt → Instructions (globalne + lokalne) → Pliki kontekstu → Historia → Output

Rodzaje instrukcji

Rodzaj Lokalizacja Ładowanie
📌 Globalne (Twój styl) ~\...\User\prompts\*.instructions.md Do KAŻDEJ rozmowy w KAŻDYM projekcie
applyTo: "**" = zawsze, "**/*.sql" = tylko SQL
📁 Projektowe (ten projekt) .github/copilot-instructions.md Tylko w TYM projekcie, do każdej rozmowy
📂 Warunkowe (per język) .github/instructions/*.instructions.md Tylko gdy plik pasuje do applyTo
💡 Globalne + Projektowe + Warunkowe = SUMUJĄ się. Konflikty? Lokalne > Globalne
⚠️ applyTo: "**" = ładuje się ZAWSZE = zajmuje context window. Używaj selektywnie!

Jak wygląda plik .instructions.md?

---                                    ← YAML start
applyTo: "**/*.sql"                    ← KIEDY
---                                    ← YAML end

# Konwencje SQL @ Allegro              ← tytuł

- Główny projekt: `sc-16612-jarvis-cloud-prod`
- Prywatny dataset: `emilia_matrejek`
- Zawsze filtruj po `_partitiondate`
- `GROUP BY ALL` zamiast numerowania
- Komentarze po polsku

## Tabele                              ← wiedza
| Metryka | Tabela           | Klucz       |
| GMV     | General.orders   | price*fx    |
| Revenue | billings.billings| ...         |

• Markdown — bez limitu długości (ale pamiętaj o context!)

• Możesz używać tabel, list, nagłówków

• Możesz używać XML tags (<conventions>...</conventions>)

• Komentarze tylko gdy logika nieintuicyjna
